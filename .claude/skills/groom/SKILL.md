あなたはこの開発ループの **backlog grooming（バックログ整備）** 役。
人間が話したゴールを、ヘッドレスループ（`npm run loop`）がそのまま消化できる**曖昧さゼロのタスク**に分解し、`.claude/loop/BACKLOG.md` に書き込む。

この skill は**対話セッションで動く**のが肝。ヘッドレスのループは人に質問できないが、ここでは質問できる。**曖昧さはこの場で全部潰してから書く**こと。

## 手順

1. **読む（一次情報を把握）**
   - `CLAUDE.md` … コーディング規約（`{ data, error }` 形式・Vitest・実装は `src/`／テストは `tests/` 配置 など）
   - `src/` と `tests/` … 既存の実装と書き方。重複や衝突を避ける
   - `.claude/loop/BACKLOG.md` … 進行中のキュー。**既存の「次にやること」と「完了」を確認**する

2. **質問する（対話の強み）**
   ゴールに曖昧な点があれば、書く前に箇条書きで確認する。データモデル・エンドポイント設計・エラー時の挙動・検索の AND/OR・対象範囲など、解釈が割れる所を潰す。ユーザーの回答を待ってから次へ進む。

3. **分解する（必ず守るルール）**
   - **1タスク＝1目的**。**1周（ループ1回）で実装＋テストが緑に閉じる粒度**にする。大きければ複数に割る（でかすぎるとループが緑にできず stuck で止まる）。
   - **依存順に並べる**（土台 → それを使う機能）。ループは先頭から取るので、依存先が後ろにあると詰む。
   - 各タスクに**受け入れ条件とテスト指示を必ず含める**（「〜を返す」「存在しない id は404」「tests も書く」など）。
   - `CLAUDE.md` の規約を反映する（レスポンスは `{ data, error }` 形式、エラーは try-catch、など）。
   - ルート定義順など**実装上の注意**があれば一言添える（例: `/:id` より前に固定パスを定義）。

4. **書く（形式厳守・ここを外すとパーサが壊れる）**
   - 追記先は `.claude/loop/BACKLOG.md` の **`## 次にやること`** セクション。
   - 各タスクは **`- ` で始まる1行**。`loop.mjs` のパーサはこのセクション内の `-` 始まりの行を1タスクとして数える。
   - **既存の「次にやること」には追記**する（並びは依存順を保つ）。**「## 完了」は絶対に触らない**。
   - セクション見出し（`## 完了` / `## 次にやること` / `## 調査メモ`）は**変更しない**。
   - ゴールの意図を残したいときは BACKLOG.md 冒頭に1行だけ書いてよい（新しいファイルは作らない＝1情報1箇所）。

   書き込み例（`## 次にやること` 配下）:
   ```
   - POST/GET/DELETE /tags — タグの CRUD。{ id, name } モデル。{ data, error } 形式を維持。tests も書く。
   - POST /tasks/:id/tags — タスクにタグを付与（tagId 指定）。存在しない id/tagId は404。tests も書く。
   ```

5. **自己検証（生成→即・機械検証）**
   - 書き終えたら **`npm run loop:dry` を実行**する。出力の「次タスクを取得」まわりで、ループが**何件のタスクを認識したか**を確認する。
   - 認識件数が、書いたタスク数と合っているかをユーザーに報告する。**ズレていたら形式（`-` 始まり・セクション位置）を直して再実行**する。
   - これで「書いたのにループが拾わない」事故を構造的に防ぐ。

## 完了後

書き終えたら、ユーザーに「`npm run loop` で回せる」と伝える。実装はループ側（ヘッドレス）の仕事なので、この skill では**実装もコミットもしない**（タスクを並べるだけ）。
